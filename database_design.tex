\section{Planned End Users and Their Needs}

\subsection{End Users}

The PARS (Personalized Apartment Ranking System) is designed for the following user categories:

\begin{enumerate}
    \item \textbf{Apartment renters} — the primary target group. Users looking for apartments to rent who want to receive personalized recommendations based on their visual and functional preferences, not just formal filters.
    
    \item \textbf{System administrators} — manage data, machine learning models, update apartment information from external sources (CIAN API).
\end{enumerate}

\subsection{User Needs}

\subsubsection{Renters}
\begin{itemize}
    \item \textbf{Personalized search:} Ability to train the system to their taste by rating apartments on three factors (beauty, price-quality, location).
    
    \item \textbf{Flexible filters:} Configuration of search parameters (city, price range, number of rooms, floor) and points of interest (workplace, gym, etc.).
    
    \item \textbf{Multiple profiles:} Ability to create multiple profiles for different scenarios (e.g., work in the city center and country house).
    
    \item \textbf{Rating history:} View previously rated apartments and ability to change ratings.
    
    \item \textbf{Recommendations:} Receive personalized apartment rankings based on trained model.
    
    \item \textbf{Profile sharing:} Ability to use pre-trained profiles from other users or share your own.
    
    \item \textbf{Quick access:} View apartment photos, travel time information to important points, links to full listings.
\end{itemize}

\subsubsection{Administrators}
\begin{itemize}
    \item \textbf{Data management:} Add, update, and delete apartment listings.
    
    \item \textbf{Monitoring:} Track model training metrics, prediction stability.
    
    \item \textbf{Maintenance:} Update caches, clean up outdated data, manage model versions.
\end{itemize}

\section{Functional Requirements and Data Restrictions}

\subsection{Functional Requirements}

\subsubsection{User and Profile Management}
\begin{enumerate}
    \item The system must register users via Telegram (by \texttt{tg\_user\_id}).
    \item User can create one or more profiles (alias). Within a single user, profile names must be unique.
    \item Each profile contains:
    \begin{itemize}
        \item Search filters (city, price range, number of rooms, floor) in JSON format.
        \item Factor weights (beauty, price-quality, location), the sum of which must equal 1.0.
        \item Exploration parameter (ε-greedy) for managing the balance between exploit and explore.
        \item Learning stage ('single' or 'pairwise').
    \end{itemize}
    \item Profile can be public (accessible to other users) or private.
    \item Public profile must have a unique \texttt{public\_slug} for access.
    \item Profile can be forked (copied) from another profile with weights and model preserved, but without rating history.
\end{enumerate}

\subsubsection{Points of Interest (POI) Management}
\begin{enumerate}
    \item User can create multiple points of interest (work, gym, etc.) with coordinates (latitude, longitude).
    \item Within a single user, POI labels must be unique.
    \item Profile can be linked to one or more POIs through the junction table \texttt{profile\_pois}.
    \item For each profile-POI relationship, maximum travel time and transport mode (masstransit, driving, walking) are specified.
\end{enumerate}

\subsubsection{Apartment Management}
\begin{enumerate}
    \item The system stores apartment listings from CIAN with a unique \texttt{cian\_id}.
    \item Each apartment contains basic information: price, area, number of rooms, floor, address, coordinates, nearest metro station.
    \item Apartment can have multiple photos, ordered by the \texttt{seq} field.
    \item For each photo, a visual embedding (CLIP) can be computed, which is stored in S3, and the link is stored in the database.
    \item Apartment can be active or deactivated.
\end{enumerate}

\subsubsection{Travel Time Calculation}
\begin{enumerate}
    \item For each pair (apartment, POI), travel time can be calculated and cached for different transport modes.
    \item Cache has TTL (\texttt{expires\_at}) for periodic updates.
\end{enumerate}

\subsubsection{Ratings and Learning}
\begin{enumerate}
    \item User can rate an apartment on three factors (1-5): beauty (\texttt{beauty}), price-quality (\texttt{price\_quality}), location (\texttt{distance\_pref}).
    \item Within a single profile, user can rate each apartment only once (uniqueness by \texttt{user\_id}, \texttt{profile\_id}, \texttt{flat\_id}).
    \item Ratings can be incomplete (nullable fields), but at least one factor must be rated, otherwise the record is marked as \texttt{skipped=true}.
    \item User can compare two apartments pairwise by selected factor (pairwise comparisons).
    \item Within a single profile, there cannot be two identical comparison pairs for one factor (uniqueness by \texttt{user\_id}, \texttt{profile\_id}, \texttt{flat\_a\_id}, \texttt{flat\_b\_id}, \texttt{factor}).
    \item \texttt{preferred\_flat\_id} in pairwise must equal either \texttt{flat\_a\_id} or \texttt{flat\_b\_id}.
\end{enumerate}

\subsubsection{Delivery Queue and View History}
\begin{enumerate}
    \item The system maintains a queue of apartments for display to each profile (\texttt{profile\_delivery\_queue}).
    \item Each apartment in the queue has a state: 'queued', 'shown', 'rated', 'skipped'.
    \item User can save or hide apartments, which should exclude them from future deliveries.
    \item The system tracks viewed apartments to exclude repeats.
\end{enumerate}

\subsubsection{Machine Learning Models}
\begin{enumerate}
    \item For each profile, a preference prediction model can be trained.
    \item Model is saved as a checkpoint (snapshot) in S3 with metrics (Kendall τ, MAE).
    \item Profile stores a reference to the last trained snapshot.
    \item The system maintains profile metrics: number of ratings, number of pairwise comparisons, current stability.
\end{enumerate}

\subsubsection{Ranking and Score Cache}
\begin{enumerate}
    \item For each pair (profile, apartment), final score can be calculated and cached.
    \item Score is calculated as a weighted sum of factor predictions: 
    $$\text{score} = w_b \cdot \hat{beauty} + w_{pq} \cdot \hat{price\_quality} + w_d \cdot \hat{distance}$$
\end{enumerate}

\subsection{Data Restrictions}

\subsubsection{Required Fields}
\begin{itemize}
    \item \texttt{users}: \texttt{tg\_user\_id} must be unique if specified.
    \item \texttt{profiles}: \texttt{cian\_filter} cannot be NULL, weights must be in range [0,1], sum of weights must be ≈ 1.0.
    \item \texttt{flats}: \texttt{cian\_id} must be unique, \texttt{active} defaults to \texttt{true}.
    \item \texttt{ratings}: at least one of the fields (\texttt{beauty}, \texttt{price\_quality}, \texttt{distance\_pref}) must be not NULL, otherwise \texttt{skipped=true}.
    \item \texttt{ratings}: ratings must be in range 1-5 if specified.
\end{itemize}

\subsubsection{Referential Integrity}
\begin{itemize}
    \item All foreign keys must reference existing records.
    \item When a user is deleted, their profiles, POIs, and ratings are cascaded deleted (or deletion restriction is set).
    \item When an apartment is deleted, photos and related records are cascaded deleted.
    \item When a profile is deleted, related ratings, delivery queue, and metrics are cascaded deleted.
\end{itemize}

\subsubsection{Business Rules}
\begin{itemize}
    \item User cannot rate the same apartment twice within one profile.
    \item Pairwise comparison cannot be made between the same apartment (flat\_a\_id ≠ flat\_b\_id).
    \item Profile cannot be forked from itself (recursive check).
    \item Public profile must have a unique \texttt{public\_slug}.
\end{itemize}

\section{Non-Functional Requirements}

\begin{enumerate}
    \item \textbf{Performance:}
    \begin{itemize}
        \item Query to get next apartment must execute in no more than 500ms.
        \item Query to get top-10 recommendations must execute in no more than 1s.
        \item Caching of travel time to reduce load on external APIs.
    \end{itemize}
    
    \item \textbf{Scalability:}
    \begin{itemize}
        \item Support for up to 10,000 users and 100,000 apartments.
        \item Indexes on frequently used fields for query optimization.
    \end{itemize}
    
    \item \textbf{Reliability:}
    \begin{itemize}
        \item Transactionality of critical operations (profile creation, rating saving).
        \item Daily database backups.
    \end{itemize}
    
    \item \textbf{Security:}
    \begin{itemize}
        \item Data personalization: user sees only their profiles and ratings.
        \item Public profiles are accessible to all, but without user personal information.
    \end{itemize}
\end{enumerate}

\subsection{Entity and Relationship Description}

\subsubsection{Main Entities}

\begin{description}
    \item[\textbf{users}] System users (Telegram users). Contains identifier, Telegram ID, username, timestamps.
    
    \item[\textbf{profiles}] User profiles (alias). Contains search filters, factor weights, learning parameters, public flag, reference to forked model.
    
    \item[\textbf{pois}] User points of interest (work, gym, etc.). Contains label, coordinates.
    
    \item[\textbf{flats}] Apartments from CIAN. Contains all apartment characteristics: price, area, address, coordinates, metro, etc.
    
    \item[\textbf{flat\_photos}] Apartment photos. Linked to apartment, contains sequence number, URL, room type.
    
    \item[\textbf{photo\_embeddings}] Photo embedding metadata. Stores link to vector in S3, model, dimensionality.
    
    \item[\textbf{ratings}] Single user ratings (1-5 on three factors).
    
    \item[\textbf{pairwise\_ratings}] Pairwise apartment comparisons by selected factor.
    
    \item[\textbf{model\_snapshots}] Trained model checkpoints with metrics.
    
    \item[\textbf{profile\_metrics}] Aggregated profile metrics.
    
    \item[\textbf{profile\_flat\_score}] Materialized cache of calculated scores for fast ranking.
\end{description}

\subsubsection{Relationships}

\begin{itemize}
    \item \texttt{users} $\rightarrow$ \texttt{profiles} (1:N) — one user can have multiple profiles.
    \item \texttt{users} $\rightarrow$ \texttt{pois} (1:N) — one user can have multiple POIs.
    \item \texttt{profiles} $\leftrightarrow$ \texttt{pois} (M:N via \texttt{profile\_pois}) — profile can be linked to multiple POIs.
    \item \texttt{flats} $\rightarrow$ \texttt{flat\_photos} (1:N) — one apartment has many photos.
    \item \texttt{flat\_photos} $\rightarrow$ \texttt{photo\_embeddings} (1:1) — one photo can have one embedding.
    \item \texttt{flats} $\leftrightarrow$ \texttt{pois} (M:N via \texttt{flat\_poi\_travel}) — travel time cache.
    \item \texttt{users} $\times$ \texttt{profiles} $\times$ \texttt{flats} $\rightarrow$ \texttt{ratings} — user ratings of apartments within a profile.
    \item \texttt{profiles} $\rightarrow$ \texttt{model\_snapshots} (1:N) — profile can have multiple model checkpoints.
    \item \texttt{profiles} $\rightarrow$ \texttt{profile\_metrics} (1:1) — one profile has one metrics record.
    \item \texttt{profiles} $\times$ \texttt{flats} $\rightarrow$ \texttt{profile\_flat\_score} — score cache for each pair.
\end{itemize}

\subsection{Multi-Valued Dependencies}

In the considered schema, no multi-valued dependencies (MVD) requiring normalization to 4NF have been found. All M:N relationships are implemented through junction tables, which corresponds to normal form.

A potential multi-valued dependency could arise if photo information were stored together with apartment information in one table, but in our schema this is separated into \texttt{flats} and \texttt{flat\_photos} with a foreign key.

\section{Database Schema Normalization}

\subsection{Normal Form Verification}

\subsubsection{First Normal Form (1NF)}

All tables are in 1NF:
\begin{itemize}
    \item All attributes are atomic (no composite values in one field).
    \item No repeating groups.
    \item Exception: the \texttt{cian\_filter} field of type JSONB contains structured data, but this is acceptable in modern DBMS for filter flexibility.
\end{itemize}

\subsubsection{Second Normal Form (2NF)}

All tables are in 2NF:
\begin{itemize}
    \item All tables have primary keys.
    \item No partial dependencies of non-key attributes on composite keys.
    \item For tables with composite keys (\texttt{flat\_poi\_travel}, \texttt{profile\_pois}, \texttt{profile\_flat\_score}, \texttt{seen\_flats}, \texttt{saved\_flats}, \texttt{hidden\_flats}), all non-key attributes depend on the full composite key.
\end{itemize}

\subsubsection{Third Normal Form (3NF)}

All tables are in 3NF:
\begin{itemize}
    \item No transitive dependencies of non-key attributes.
    \item All non-key attributes depend only on the primary key.
\end{itemize}

\subsubsection{Boyce-Codd Normal Form (BCNF)}

All tables are in BCNF:
\begin{itemize}
    \item For each functional dependency $X \rightarrow Y$, where $X$ is not a superkey, it has been verified that no such dependency exists, or it is trivial.
    \item All determinants are potential keys.
    
    \textbf{Example verification for \texttt{ratings}:}
    \begin{itemize}
        \item Dependency $(\texttt{user\_id}, \texttt{profile\_id}, \texttt{flat\_id}) \rightarrow \texttt{id}$ — determinant is a key, OK.
        \item Dependency $\texttt{id} \rightarrow$ all other attributes — determinant is a key, OK.
        \item No other non-trivial dependencies where determinant is not a key.
    \end{itemize}
\end{itemize}

\subsubsection{Fourth Normal Form (4NF) — Optional}

All tables are in 4NF:
\begin{itemize}
    \item No multi-valued dependencies not following from keys have been found.
    \item All M:N relationships are implemented through junction tables.
\end{itemize}

\subsection{Example of Problems Due to Undernormalization}

If we were to combine the \texttt{flats} and \texttt{flat\_photos} tables into one table with repeating groups:

\textbf{Problematic schema:}
\begin{lstlisting}[language=SQL]
CREATE TABLE flats_with_photos (
    flat_id BIGINT PRIMARY KEY,
    cian_id BIGINT UNIQUE,
    price_rub INTEGER,
    -- ... other apartment fields
    photo_url_1 TEXT,
    photo_url_2 TEXT,
    photo_url_3 TEXT,
    -- ... maximum N photos
    photo_seq_1 INTEGER,
    photo_seq_2 INTEGER,
    -- ...
);
\end{lstlisting}

\textbf{Problems:}
\begin{enumerate}
    \item \textbf{Photo count limitation:} If an apartment has more photos than fields provided, information is lost or requires schema changes.
    
    \item \textbf{Storage redundancy:} For apartments with fewer photos, most fields will be NULL, increasing table size.
    
    \item \textbf{Query complexity:} Finding all photos requires checking multiple fields (photo\_url\_1, photo\_url\_2, ...).
    
    \item \textbf{Atomicity violation:} Fields photo\_url\_N and photo\_seq\_N are logically related but stored separately, which can lead to inconsistency.
    
    \item \textbf{Update complexity:} Adding or removing a photo requires updating multiple fields.
\end{enumerate}

\textbf{Solution through normalization:} Separation into \texttt{flats} and \texttt{flat\_photos} with a foreign key eliminates all these problems.

\section{SQL DDL Script for Database Creation}

\begin{lstlisting}[language=SQL, caption=DDL script for creating PARS: Personalized Apartment Ranking System database]
-- ============================================================
-- PARS: Personalized Apartment Ranking System Database Schema
-- PostgreSQL 15+
-- ============================================================

-- Creation of sequences for auto-increment (bigserial uses them automatically)

-- Table users
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    tg_user_id BIGINT,
    username TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE UNIQUE INDEX ix_users_tg_user_id ON users (tg_user_id) WHERE tg_user_id IS NOT NULL;

-- Table flats (independent, created before profiles to avoid circular dependencies)
CREATE TABLE flats (
    id BIGSERIAL PRIMARY KEY,
    cian_id BIGINT NOT NULL UNIQUE,
    url TEXT NOT NULL,
    city TEXT NOT NULL,
    address TEXT,
    lat NUMERIC(10,7),
    lng NUMERIC(10,7),
    price_rub INTEGER,
    rooms INTEGER,
    area_sqm NUMERIC(7,2),
    floor INTEGER,
    floors_total INTEGER,
    building_year INTEGER,
    material TEXT,
    nearest_metro TEXT,
    metro_distance_m INTEGER,
    published_at TIMESTAMPTZ,
    fetched_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    active BOOLEAN NOT NULL DEFAULT TRUE,
    deactivated_at TIMESTAMPTZ
);

CREATE INDEX idx_flats_active ON flats (active);
CREATE INDEX idx_flats_city_published ON flats (city, published_at);

-- Table profiles
CREATE TABLE profiles (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL REFERENCES users(id),
    alias TEXT NOT NULL,
    city TEXT NOT NULL,
    cian_filter JSONB NOT NULL,
    weight_beauty NUMERIC(5,3) NOT NULL DEFAULT 0.50,
    weight_price_quality NUMERIC(5,3) NOT NULL DEFAULT 0.30,
    weight_distance NUMERIC(5,3) NOT NULL DEFAULT 0.20,
    epsilon_explore NUMERIC(5,3) NOT NULL DEFAULT 0.20,
    stage TEXT NOT NULL DEFAULT 'single' CHECK (stage IN ('single', 'pairwise')),
    is_public BOOLEAN NOT NULL DEFAULT FALSE,
    public_slug TEXT UNIQUE,
    forked_from_profile_id BIGINT REFERENCES profiles(id),
    last_trained_snapshot_id BIGINT, -- FK will be added after creating model_snapshots
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT uix_profile_user_alias UNIQUE (user_id, alias),
    CONSTRAINT chk_weights_sum CHECK (
        ABS(weight_beauty + weight_price_quality + weight_distance - 1.0) < 0.001
    )
);

CREATE INDEX idx_profiles_user_id ON profiles (user_id);
CREATE INDEX idx_profiles_public_slug ON profiles (public_slug) WHERE public_slug IS NOT NULL;

-- Table pois
CREATE TABLE pois (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    label TEXT NOT NULL,
    lat NUMERIC(10,7) NOT NULL,
    lng NUMERIC(10,7) NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT uix_pois_user_label UNIQUE (user_id, label)
);

CREATE INDEX idx_pois_user_id ON pois (user_id);

-- Table profile_pois (junction M:N)
CREATE TABLE profile_pois (
    profile_id BIGINT NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
    poi_id BIGINT NOT NULL REFERENCES pois(id) ON DELETE CASCADE,
    max_travel_min INTEGER NOT NULL DEFAULT 45,
    mode TEXT NOT NULL DEFAULT 'masstransit' CHECK (mode IN ('masstransit', 'driving', 'walking', 'cycling')),
    priority INTEGER NOT NULL DEFAULT 1,
    PRIMARY KEY (profile_id, poi_id)
);

-- Table flat_photos
CREATE TABLE flat_photos (
    id BIGSERIAL PRIMARY KEY,
    flat_id BIGINT NOT NULL REFERENCES flats(id) ON DELETE CASCADE,
    seq INTEGER NOT NULL,
    url TEXT NOT NULL,
    room_type TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT uix_flat_photos_flat_seq UNIQUE (flat_id, seq)
);

CREATE INDEX idx_flat_photos_flat_id ON flat_photos (flat_id);

-- Table photo_embeddings
CREATE TABLE photo_embeddings (
    id BIGSERIAL PRIMARY KEY,
    photo_id BIGINT NOT NULL REFERENCES flat_photos(id) ON DELETE CASCADE,
    storage_uri TEXT NOT NULL,
    model TEXT NOT NULL DEFAULT 'CLIP-ViT-B/32',
    dim INTEGER NOT NULL DEFAULT 512,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT uix_photo_embeddings_photo_id UNIQUE (photo_id)
);

-- Table flat_poi_travel (travel time cache)
CREATE TABLE flat_poi_travel (
    flat_id BIGINT NOT NULL REFERENCES flats(id) ON DELETE CASCADE,

\section{Grouping Queries into Transactions}

\subsection{Transaction 1: Create Profile with POI}

\begin{lstlisting}[language=SQL, caption=Atomic creation of profile and POI linking (using WITH to pass ID)]
BEGIN;

-- Create profile and get its ID through CTE
WITH new_profile AS (
    INSERT INTO profiles (user_id, alias, city, cian_filter, weight_beauty, weight_price_quality, weight_distance)
    VALUES (
        1,
        'work_45min',
        'Moscow',
        '{"min_price": 30000, "max_price": 80000, "rooms": [1, 2]}'::jsonb,
        0.5,
        0.3,
        0.2
    ) RETURNING id
),
new_poi AS (
    INSERT INTO pois (user_id, label, lat, lng)
    VALUES (1, 'Work', 55.7558, 37.6173)
    ON CONFLICT (user_id, label) DO UPDATE SET lat = EXCLUDED.lat, lng = EXCLUDED.lng
    RETURNING id
)
INSERT INTO profile_pois (profile_id, poi_id, max_travel_min, mode)
SELECT np.id, npoi.id, 45, 'masstransit'
FROM new_profile np, new_poi npoi;

-- Create metrics record (use subquery to get last created profile)
INSERT INTO profile_metrics (profile_id, ratings_count, pairwise_count)
SELECT id, 0, 0
FROM profiles
WHERE user_id = 1 AND alias = 'work_45min'
ON CONFLICT (profile_id) DO NOTHING;

COMMIT;

\end{lstlisting}

\textbf{Note:} In real implementation (e.g., in PL/pgSQL or through ORM) variables are used for storing ID. In pure SQL CTE (Common Table Expressions) or subqueries are used.

\section{Conclusion}

This document presents complete database design for the PARS: Personalized Apartment Ranking System — a personalized apartment ranking service. 

The database is designed taking into account all functional requirements, normalized to BCNF (and optionally to 4NF), contains necessary integrity constraints and indexes for performance optimization.

The schema supports:
\begin{itemize}
    \item Multiple user profiles with flexible filters
    \item Rating system and pairwise comparisons for training ML models
    \item Caching of travel time and model predictions
    \item Public profiles and forking capability
    \item Delivery queue and view history
\end{itemize}

All queries are grouped into transactions to ensure atomicity of critical operations.
