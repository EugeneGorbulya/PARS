# 📝 Roadmap & TODO: SmartRentRanker

Этот план составлен на основе архитектурной документации проекта. Задача — создать рекомендательную систему аренды жилья с обучением на предпочтениях пользователя.

---

## 🏗 Этап 1: Инфраструктура и База Данных
*Подготовка окружения для хранения данных и запуска сервисов.*

- [ ] **Настройка репозитория**
    - [ ] Инициализировать git, настроить `.gitignore` (Python, IDE).
    - [ ] Выбрать менеджер зависимостей (Poetry или `requirements.txt`).
- [ ] **Docker Compose (Базовый слой)**
    - [ ] Написать `docker-compose.yml` с сервисами:
        - [ ] `PostgreSQL 15` (основное хранилище).
        - [ ] `Redis` (кэш и брокер сообщений для Celery).
        - [ ] `MinIO` (S3-compatible хранилище для фото и моделей).
- [ ] **Проектирование БД (SQLAlchemy / Tortoise ORM)**
    - [ ] Реализовать модели (схема из раздела 10):
        - [ ] `users` (telegram_id, created_at).
        - [ ] `profiles` (alias, filters JSON, weights JSON, state, active_model_id).
        - [ ] `flats` (cian_id, price, area, address, coords, features JSON).
        - [ ] `flat_photos` (flat_id, url, s3_path, vector_id).
        - [ ] `ratings` (single-label: beauty, price, distance 1-5).
        - [ ] `pairwise_ratings` (A vs B comparison).
        - [ ] `pois` и `flat_poi_travel` (кэш времени в пути).
    - [ ] Настроить миграции (Alembic).

## 🧠 Этап 3: ML Core (Обучение и Инференс)
*Сердце системы. Обработка изображений и ранжирование.*

- [ ] **Визуальные эмбеддинги**
    - [ ] Подключить CLIP (OpenAI или RuCLIP).
    - [ ] Скрипт для генерации эмбеддингов для всех скачанных фото.
    - [ ] Сохранение эмбеддингов (векторная БД или `pgvector`, либо файлы в S3 + ссылки в БД).
- [ ] **Модель предпочтений (PyTorch)**
    - [ ] Реализовать архитектуру "Туловища" (Backbone) + 3 головы (`beauty`, `price_quality`, `distance`).
    - [ ] Реализовать Dataset и Dataloader, умеющий работать с таблицами `ratings` и `pairwise_ratings`.
    - [ ] **Режим обучения Single-label:** Loss для оценок 1-5 (MSE или CrossEntropy).
    - [ ] **Режим обучения Pairwise:** Loss для сравнений (Hinge Loss / Logistic Loss).
- [ ] **Exploration Logic**
    - [ ] Реализовать функцию `get_exploration_candidates()`: выбор "неуверенных" или новых квартир (ε-greedy стратегия).
    - [ ] Реализовать MMR (Maximal Marginal Relevance) для разнообразия выдачи.

*Прослойка между Ботом, ML и БД.*

- [ ] **API Endpoints**
    - [ ] `POST /profile` — создание/обновление профиля и весов.
    - [ ] `GET /next_flat` — получение следующей квартиры для оценки (с учетом логики Exploration).
    - [ ] `POST /rate` — сохранение оценки (1-5).
    - [ ] `POST /compare` — сохранение выбора в парном сравнении.
    - [ ] `GET /top` — выдача персонального топа квартир (Inference по всем активным объявлениям).
- [ ] **Background Tasks (Celery/Arq)**

### 💡 Полезные заметки из документации
* **Обучение:** Не забыть про валидацию по времени (hold-out), чтобы модель не переобучалась на старых данных.
* **Exploration:** В начале обучения `epsilon` (доля случайных/новых квартир) должна быть высокой (0.2-0.3), затем снижаться.
* **S3:** Не хранить картинки в базе данных, только пути.
